# screamer v1.0 readiness assessment

An audit of what stands between screamer 0.12.0 and a v1.0 that commits to a stable
API. Four dimensions were reviewed: functionality, API design, function
documentation, and example notebooks. The findings are organized by the only axis
that matters for a version freeze: whether a fix is **breaking to make later**
(so it must happen before 1.0) or **additive** (safe for any 1.x release).

The headline: the library is substantially complete and internally consistent. The
Rolling family (55 ops), the `Ew*` family, the backtest grid (8 engines), and the
microstructure suite are all v1.0-ready in substance. What remains is a small set of
API-consistency fixes that are cheap now and impossible-without-deprecation later, a
few real documentation-correctness bugs, and onboarding polish. The large
functionality items are genuinely additive and belong on the 1.x roadmap.

## Tier 1: API-freeze blockers (cheap now, breaking after 1.0)

These are the real gating items. Each is a small change today and a breaking change
once the API is frozen.

1. **Parameter-name inconsistencies on windowed/forgetting ops.** Three ops deviate
   from the library's own conventions:
   - `EwKyleLambda(span=20.0)` accepts only `span`, not the
     `com`/`span`/`halflife`/`alpha` mutex every other `Ew*` op uses
     (`screamer/microstructure.py`). Adopt the family constructor.
   - `Propagator(window=20, ...)` uses `window`, the sole windowed op not using
     `window_size` (`bindings/bindings_micro.cpp`).
   - `TRIX(span=14)` uses `span` where the other windowed rolling indicators use
     `window_size`.
   These are one-line binding fixes now; renaming a keyword after 1.0 breaks user
   code.

2. **`alpha` names three unrelated things.** `EwMean` `alpha` is a forgetting factor
   in (0,1); `RollingCVaR` `alpha` is a tail probability; `HawkesIntensity` `alpha`
   is an excitation intensity (any positive float). One keyword, three incompatible
   meanings. Decide before freeze: rename the non-forgetting uses (for example
   `RollingCVaR(level=)`, `HawkesIntensity(excitation=)`) or accept and document the
   overload deliberately.

3. **Regression output column order is inconsistent.** `BayesianRegression` emits
   `[pred_mean, pred_std, slope, intercept]`; `RollingLinearRegression` emits
   `[slope, intercept, r2, stderr]`. Slope is column 0 in one and column 2 in the
   other. They serve different purposes (a forecaster vs a fitter), so this may be
   acceptable, but it is a convention to settle consciously before it is frozen.

4. **The `output=` integer selector means different things per op.** `Hampel` and
   `ImpulseClip` use modes 0-2, `RollingSigmaClip` 0-3, `RollingOU` 0-3, each with
   different semantics under the same parameter name. Consider named modes, or at
   least document each op's range on its page.

5. **Namespace hygiene.** `install_param_capture` and `screamer_bindings` leak into
   `dir(screamer)`. Hide the internal names before users start importing them.

## Tier 2: correctness bugs (should fix for a credible v1.0)

Not breaking, but wrong, and a v1.0 should not ship with wrong behavior or docs.

1. **Contradictory NaN documentation on 15 rolling pages.** Legacy inline text ("NaN
   values are not handled natively and should be preprocessed") sits in the same
   page as the generated footnote stating `Policy: ignore`. The pages contradict
   themselves and the actual behavior. Delete the stale sentence.

2. **NaN slips through the forgetting-factor guard.** `EwMean(alpha=float('nan'))` is
   silently accepted and produces all-NaN output, because `alpha <= 0 || alpha >= 1`
   is false for NaN. This affects every `Ew*` op and `BayesianRegression`. Add an
   `isfinite` check to the family-wide guard so garbage input fails loudly.

3. **Wrong comment** in `bindings_misc.cpp`: it says CumSum/CumProd/CumMax/CumMin
   "NaN propagates", but all four implement the standard "ignore" policy.

4. **`UltimateOscillator`** frontmatter has empty `description:` fields for
   `period1`/`period2`/`period3`, the only op with undocumented parameters.

## Tier 3: documentation example quality (raises the bar)

1. **37 pages have static-only examples with no rendered output.** The entire
   Expanding family (10 pages) shows only a 2-4 line snippet; the signal family (5
   pages) shows filter code with no data or plot; 14 more rolling pages likewise. A
   v1.0 doc set should have a runnable, rendered example on every page.

2. **7 `Ew*` pages have empty `### Usage Example and Plot` headings** (plots render
   further down; the heading is stale/cosmetic).

3. **`see_also` is sparse** (27 of 226 pages). The 61-page rolling family has none,
   despite natural clusters (BollingerBands/Keltner/Donchian; the despiking filters;
   RollingMax/Min/Range/MinMax).

## Tier 4: onboarding (a v1.0 invites new users)

1. **No notebook covers uncertainty.** `KalmanFilter` and `BayesianRegression` both
   ship with reference pages but appear in none of the 16 notebooks.
   `BayesianRegression` is the library's only calibrated-interval operator and is
   invisible to anyone who has not read the catalog. A "filtering and forecasting
   with uncertainty" notebook is the fix.

2. **No cross-catalog "how to choose an operator" guide.** With 222 ops a user who
   knows their task ("despike outliers", "fit a trend") has no guided path. The
   `choosing_a_backtest_engine` page is the pattern to generalize.

3. **Family coverage gaps in existing notebooks.** The signal-processing notebook
   covers only one of three despiking ops (missing `Hampel`, `RollingSigmaClip`) and
   only the low-pass Butterworth. The pairs/regression family (`RollingBeta`,
   `RollingSpread`, `RollingAlpha`, `RollingOU`) has no worked example.

4. **No installation/prerequisites page** beyond a single `pip install` line (no
   Python version floor, no dev-install path).

## Tier 5: functionality (additive, safe for 1.x)

None of these block a freeze; each extends the API without breaking it. They are the
1.x roadmap, most already sketched in existing specs.

- **Expose the `KalmanFilter` posterior variance** (it computes it and discards it).
- **Multivariate (k-predictor) regression**, screamer's first N-series joint op.
- **Conjugate online Bayesian estimators**: Beta-Bernoulli probability, Gamma-Poisson
  rate, Bayesian online changepoint detection.
- **`Delay` inside a live Pipeline feeding a merge** (the documented reorder-buffer
  limitation).
- **ExpandingCorr / ExpandingBeta / ExpandingCov** (the Expanding family has no
  cross-series ops).
- **EW `ignore_na=False`** decay-aware NaN, and **Resample per-column fill**, both
  noted as out-of-scope in their current docs.

### Two architectural gaps that need a decision before freeze

These are additive to implement, but the *shape* of the answer touches the frozen
API, so decide the intended contract now even if the implementation waits:

- **Multi-input polymorphism.** An operator with more than one input returns a plain
  list, not a lazy iterator, when fed iterables, and the general N-input/M-output
  case raises `NotImplementedError` (documented in `polymorphic_api.md`). If the v1.0
  promise is "every operator is polymorphic including lazy streaming", this gap
  contradicts it. Decide: is eager-for-multi-input the committed contract (document
  it as intended), or must lazy multi-input land before the freeze?
- **The `Delay` live-merge limitation.** Same question: is it an accepted v1.0
  constraint (documented) or a pre-freeze fix?

## Recommended path to v1.0

1. A focused **API-freeze hardening pass**: Tier 1 (the consistency fixes) plus Tier
   2 (the correctness bugs). This is the only work that is genuinely gated by the
   freeze, and it is small.
2. Settle the **two architectural decisions** above and write the intended contract
   into the docs, whether or not the implementation lands in 1.0.
3. A **documentation quality pass**: Tier 3 (runnable examples on the 37 bare pages)
   and Tier 4 (an uncertainty notebook and a choose-an-operator guide). This is what
   makes a v1.0 credible to a newcomer, and it is the larger effort.
4. Ship the Tier 5 functionality across 1.x as additive releases.
